## D. Slice F1 后端沙箱架构

1. 安装 server dependency `alipay-sdk@4.14.0`，只在服务端使用。
2. 新建配置与 adapter/service：私钥只能来自 `ALIPAY_PRIVATE_KEY` 或 `ALIPAY_PRIVATE_KEY_FILE`（文件优先级规则需明确）；支付宝公钥同理；任何日志、响应、测试快照不得出现密钥。`PAYMENT_MODE=mock|alipay_sandbox` 默认 `mock`；本切片禁止 production mode。
3. `alipay_sandbox` checkout：前端只传 `planId='pro'` + idempotencyKey；服务端从 DB `plans.price_fen` 取金额，创建 DB order，再用 SDK `pageExec('alipay.trade.page.pay', ...)` 生成表单/跳转载荷。禁止信任客户端金额。
4. 保留 `mock` 模式现有本地体验；返回结构增加明确 `paymentMode`。不要让 mock 状态与真实 DB 权益混淆。
5. notify 路由在全局 JSON parser 前使用 `express.urlencoded({extended:false})`；先验签，再核对 app_id、seller_id、out_trade_no、total_amount（严格转分）、trade_status，仅 `TRADE_SUCCESS/TRADE_FINISHED` 调原子 RPC。成功 commit 后返回纯文本 `success`，其他 fail closed，且不泄露内部错误。
6. `POST /billing/orders/:id/reconcile`：owner + 基本限频/幂等，调用 `alipay.trade.query`，使用同一支付应用服务确认；同步 return 不授予权益。
7. 结果页在 sandbox mode 显示“等待服务端确认”，轮询/查询订单；只有服务端订单 status=paid 才显示支付成功与 Pro 生效。取消/同步 return 不改权益。
8. `/api/billing/plans`、`/api/me/entitlements` 在 sandbox mode 读数据库现有 plans/subscriptions；mock mode 可保留内存数据。避免新增第三份套餐常量；将 app.ts 静态 plans 路由收敛到 billing router/service。
9. `.env.example` 只加变量名与占位说明，绝不写真实值；默认 mock 确保无支付宝配置时现有应用仍可运行。
