# Bad Case 审阅包使用指南（v2.1）

## 1. 进入条件

1. 使用 `super_admin` 账号登录。
2. 打开管理后台，在模型/质量区域找到“Bad Case 审阅包”。
3. 普通 `admin` 看不到或收到 403 属于正确权限表现；详情权限由服务端控制，不能用前端隐藏代替鉴权。
4. 审阅包主要由 v2.1 上线后的新生成任务自动创建。旧任务没有生成时快照时会显示 `legacy_unavailable`，系统不会拿当前 Prompt、规则或知识冒充历史版本。

## 2. 推荐审阅顺序

1. 在列表按状态、责任团队筛选，打开一条审阅包。
2. 看“概览”：确认触发原因、分数、严重度、数据 Owner、责任团队和当前处理人。
3. 看“样本”：对照原始需求、各平台文案、诊断、审核和评分，确认问题是否真实存在。
4. 看“Trace”：定位失败阶段、模型、耗时和错误分类。Trace 是脱敏运行轨迹，不包含模型思维链。
5. 看“验收”：比较每条标准的实际值、期望值和版本；`not_evaluated` 表示现有数据不足，不能当作通过。
6. 看“工件”：核对生成当时的 Prompt、规则、知识和模型策略快照。
7. 看“Findings”：逐条判断并填写处置结果和备注。
8. 最后看“审计”：确认指派、状态、分析、处置和提案行为均留下记录。

## 3. 指派与状态

责任团队建议：

- `content_prompt`：提示词、语气、结构、卖点表达。
- `knowledge_rules`：知识库、规则、节日/案例引用。
- `model_provider`：模型返回、超时、限流或供应商错误。
- `backend_platform`：接口、持久化、任务状态、权限和遥测。
- `frontend_experience`：页面呈现、交互或客户端状态。
- `unassigned`：尚未确认归属。

推荐状态流：

`open` → `triaging` → `in_progress` → `resolved`

- `wont_fix`：确认问题但决定不修，必须写清原因。
- `duplicate`：与另一条 Case 重复，备注中应写关联 Case ID。
- 终态重新打开时回到 `open`、`triaging` 或 `in_progress`。

## 4. Finding 人工处置

- `confirmed`：证据充分，问题成立。
- `false_positive`：检测器误报，输出本身没有该问题。
- `accepted_risk`：问题成立，但当前业务接受风险。
- `needs_data`：样本、Trace 或标准不足，暂不能判断。
- `resolved`：修复后复测通过。

处置备注至少写三件事：判断依据、下一动作、复测方式。不要只写“已处理”。

## 5. 运行 DeepSeek 分析

### 5.1 它具体做什么

系统先重跑确定性验收规则，保留“分数低于阈值、生成失败、输出不完整、红线失败”等可核对证据；随后只把当前样本、文案变体、规则结果、已有 Finding 和脱敏模型轨迹发给 DeepSeek。DeepSeek 返回：

- 诊断：问题可能发生在哪里、为什么；
- 修复建议：建议人工审阅哪类 Prompt、规则、知识或系统实现；
- 建议 Owner、置信度；
- 一段审阅包摘要。

DeepSeek 不会获得密钥、Cookie、JWT、邮箱、原始 provider payload、模型思维链或原始错误正文。它的输出只写入 Finding 的“AI 诊断/修复建议”，不会自动改配置、发布规则或关闭 Case。

### 5.2 按钮和状态含义

| 页面显示 | 含义 | 你要做什么 |
| --- | --- | --- |
| `运行 DeepSeek 分析` | 尚未按当前 AI 分析版本运行 | 点击一次并等待结果提示 |
| `分析中…` | 当前浏览器正在等待服务端完成 | 不要重复点击或关闭页面 |
| `分析处理中` | 服务端记录为 pending | 稍后刷新；长时间不恢复时查看 API 日志 |
| `检查 DeepSeek 分析` | 状态为 completed | 点击可检查当前版本；已经运行过会明确提示“无需重复” |
| `analysis_unavailable` | 模型、网络、密钥或响应格式暂不可用 | 规则 Findings 仍然有效；按提示稍后重试 |

旧版本中 `completed` 只表示确定性规则执行完成。v2.1 当前实现会检查带版本的 `analysis_completed` 审计事件；没有 DeepSeek 版本标记的旧记录可升级运行一次，不会被旧状态永久挡住。

### 5.3 正确操作顺序

1. 先看“样本”“Trace”“验收”，确认输入与规则证据合理。
2. 点击“运行/检查 DeepSeek 分析”，等待绿色结果条；不要把按钮文字变化当成完成证据。
3. 打开“Findings”，选中每一条；蓝色框中的“AI 诊断”和“修复建议”是模型意见。
4. 对照样本、Trace、criterion ref 和工件快照判断建议是否成立。
5. 填写 Finding 处置和备注。AI 建议错误时选择 `false_positive` 或在备注里说明错误点。
6. 需要改工件时才创建“待审提案”；提案仍需人工审核和后续真实实施。

### 5.4 成功与失败怎么判断

- 成功：页面明确显示“DeepSeek 分析完成”，并给出建议数和 Finding 总数；审计里新增 `analysis_requested` 与 `analysis_completed`。
- 当前版本已经分析：页面明确显示“已经完成，无需重复运行”；这不是无反应。
- 暂不可用：页面会显示安全错误分类，例如 `timeout`、`authentication`、`rate_limited`、`invalid_response`；不会暴露供应商原始错误。
- 建议数为 0：表示模型没有为现有确定性 Finding 生成通过校验的建议，不代表“任务没有问题”；仍需人工看规则证据。

## 6. 创建待审工件提案

只有工件状态为 `available` 时才能创建。选择 Prompt、规则、知识或模型策略后，编辑白名单 JSON Patch，并填写提案理由。

界面的默认安全示例：

```json
{"ops":[{"op":"replace","path":"/templates/0/version","value":"2.1"}]}
```

其他工件示例（每次只提交其中一个对象）：

```json
{"ops":[{"op":"replace","path":"/version","value":"2.1"}]}
{"ops":[{"op":"replace","path":"/calendar/datasetVersion","value":"2.1"}]}
{"ops":[{"op":"replace","path":"/policyVersion","value":"2.1"}]}
```

三行依次对应规则、知识、模型策略；加上前一个 Prompt 示例共四类。实际使用时只保留与当前工件类型对应的一行，并将值改成真实建议值。

服务端会重新读取数据库中的生成时快照，核对哈希、路径白名单、值类型、敏感字段和是否产生有效差异。提案只进入 `pending_review`，不会自动修改或发布 Prompt、规则、知识库或模型策略。

## 7. 何时可以关闭 Case

同时满足以下条件后再改为 `resolved`：

1. 主要 Findings 已有人工处置；
2. 责任团队已完成真实修复或明确无需代码修改；
3. 用原样本和至少一个相邻样本复测；
4. 对应验收标准通过；
5. 没有新回归，审计记录完整。

## 8. 诊断面板怎么看

- 问题分类：问题主要发生在哪个环节。
- 同类复发：是否重复出现相似 Bad Case。
- 确认率/误报率：自动检测本身是否可靠。
- 验收覆盖：哪些失败能被现有标准捕获，哪些仍是盲区。
- 解决时长：从发现到关闭花了多久。
- Token/人民币：只有供应商返回官方 usage 且命中版本化价格表时才计算；不可用不等于零成本。

## 9. 当前 staging 限制

截至 2026-07-22，数据库结构已进入 staging，但真实角色/API/浏览器闭环和最终人工验收尚未完成。因此本指南描述的是 v2.1 已实现合同，不代表生产已上线。

新增 DeepSeek 诊断还需要在 staging 使用真实 `super_admin` 完成一次：旧 `completed` 升级、首次成功、幂等复点、供应商失败态、审计顺序和 Finding 建议显示。该步骤会调用真实模型并写 staging 审计数据，必须单独获批后执行。

## 10. 一张单怎么审完（快速清单）

- [ ] 触发原因、分数、严重度与样本一致。
- [ ] Trace 已定位到具体阶段；不可用时已注明原因。
- [ ] 失败标准有 evidence refs、criterion refs 和生成时工件 refs。
- [ ] DeepSeek 建议与证据相符，没有把推测写成事实。
- [ ] Owner 团队和人员已指派，原因可追溯。
- [ ] 每条 Finding 已保存人工处置和复测方法。
- [ ] 需要修改时只创建待审提案，没有直接发布。
- [ ] 原样本和相邻样本复测通过后才把 Case 改为 `resolved`。
