# 77港话 PRD/SDD 相比原 comprehensive-spec-v2 的新增内容

> 对比基准：`docs/comprehensive-spec-v2.md`（1084 行）
> 对比对象：`docs/77港话社媒文案器-PRD-SDD.md`（~450 行，更密）

## 一、全新章节（旧 spec 完全没有的）

### 1. 目标用户与使用场景（§二）

旧 spec 只有功能列表，没有「人」。

| 新增内容 | 说明 |
|---------|------|
| **用户画像 1：Amy**（品牌社媒运营经理，28-35 岁） | 含年龄、角色、日常工作、4 个痛点、JTBD、选择原因 |
| **用户画像 2：Ken**（中小企业老板/一人营销部，35-50 岁） | 含年龄、角色、日常工作、4 个痛点、JTBD、选择原因 |
| **3 个叙事场景** | ① Amy 从总部素材到多平台发布、② Ken 父亲节借势、③ Amy 竞品监测+差异化迭代 |

**价值**：开发者在读功能列表之前先理解「谁在用、为什么用」——避免做出「技术上正确但用户不需要」的功能。

### 2. 核心用户动线（§三）—— 3 个 Mermaid 流程图

旧 spec 完全没有流程图。

| 流程图 | 覆盖内容 |
|--------|---------|
| **主流程**：生成港式文案 | 从输入原文 → 判断语言 → 判断文案类型 → toggle 决策 → 生成 → 审核 → 迭代，完整 20+ 节点 |
| **异常分支**：生成失败 | 超时/JSON 解析失败/网络错误 → 各自的降级路径 → 重试 |
| **灵感辅助流程**：使用灵感面板 | 4 个 Tab 各自的数据加载→展示→用户操作→注入 prompt 的闭环 |

**价值**：开发者可以直接据此实现状态机；测试人员可以按路径写 test case。

### 3. 交互细节 + 状态清单 + 边界条件（§六）

旧 spec 只说「这个功能做什么」。PRD 对每个核心功能补全了 PRD Writer 规范要求的 3 类细节：

**每个功能现在都有：**

| 规范项 | 以 A1「原文诊断」为例 |
|--------|---------------------|
| **交互细节表**（4 行：操作反馈/危险确认/空状态/失败引导） | 按钮 spinning → 「生成中...」；原文为空时 disabled + tooltip；诊断失败不阻塞生成 |
| **状态清单表**（6 行：默认/就绪/加载中/成功/部分失败/完全失败） | 每行含状态名、触发条件、UI 表现、用户可执行操作 |
| **边界条件表**（6 行：空/超长/网络异常/无权限/并发/格式不符） | 2000 字上限、重试按钮、按钮 loading 期间 disabled |

旧 spec 里这些信息要么不存在，要么散布在 prompt 描述里需要读者自己推断。

**价值**：AI/开发者不需要「猜」异常情况怎么处理——表里写死了。

### 4. 文案规范（§十一）—— 30+ 个 UI 位置的精确文案

旧 spec 完全没有。旧代码里的 UI 文案是开发时随手写的，没有统一规范。

| 规范内容 | 数量 |
|---------|------|
| 产品整体风格基调定义 | 1 条（亲切友好 + 简洁直接） |
| 终端用户可见的 UI 文案 | 30+ 条（placeholder / toast / 按钮 / 错误提示 / 空状态 / toggle 说明 / Tab 标签……） |
| 每条文案的格式要求 | 动词开头、原因+下一步、引导性 |

**价值**：开发者不需要自己编 UI 文字；所有文案已经风格统一、措辞一致。

### 5. 非功能性需求（§十二）

旧 spec 完全没有。

| 规范项 | 内容 |
|--------|------|
| **性能要求** | 5 个指标的目标值（首屏 <2s、生成 <30s、审核 <10s、快速检查 <100ms、灵感检索 <5s） |
| **权限控制** | 当前无登录；后续商业化分层策略 |
| **兼容性** | 浏览器版本、分辨率下限、设备优先级、OS |
| **数据安全** | 原文不持久化、localStorage 不上传、API Key 不暴露、竞品数据仅供研究 |
| **数据存储** | localStorage 上限、后端无持久化（当前）、语料库 JSON/SQLite（Phase 3） |

**价值**：上线前的 checklist——运维和合规的人可以直接用。

### 6. 错误码体系（§九.3）

旧 spec 没有统一的错误处理规范。PRD 定义了：

```ts
interface ApiError {
  error: string;           // 人类可读
  code: string;            // 机器可读：INVALID_INPUT | AI_TIMEOUT | AI_PARSE_ERROR | AI_UNAVAILABLE | RATE_LIMITED | EXTERNAL_ERROR | INTERNAL_ERROR
  retryable: boolean;      // 是否可重试
  suggestion?: string;     // 给用户的下一步建议
}
```

**价值**：前端可以根据 `code` 做统一错误处理，而不是每个接口各自 try-catch。

### 7. 待确认问题（§十五）—— 6 个决策点

旧 spec 的问题是分散在各处。PRD 汇总为 6 个待产品决策者确认的问题：
- 商业化路径
- CantoneseLLM 备选方案
- 竞品检索 ToS 合规
- 用户反馈机制
- 多语言扩展
- 语料库人工标注资源

## 二、原有章节被大幅增强的内容

### 8. 关键页面布局（§四）—— 从 1 张图变为 2 张图 + 布局决策说明

旧 spec 只在 delta summary 里有一张 Phase 2 后的 mockup。

PRD 新增：
- **当前 MVP 布局线框图**（展示已上线界面的实际结构）
- **Phase 2 完整布局线框图**（展示所有 🆕 组件的位置：toggle、日期、竞品搜索、灵感面板、CTA 评分）
- **布局决策说明**（为什么面板放 Center 下方而不是 Left Sidebar；为什么 toggle 紧邻创作自由度）

### 9. 功能清单（§五）—— 从线性表格变为树状结构 + 模块化

旧 spec：22 项优化方向排列在表格里，按影响力排序。

PRD：重构为 **8 个模块（A-H）× 树状结构**，每个模块标注优先级（🔴🟡⚪）：
- 模块 A：文案生成（10 子功能）
- 模块 B：审核评分（5 子功能）
- 模块 C：Prompt 增强（5 子功能，待建）
- 模块 D：灵感参考面板（7 子功能，待建）
- 模块 E：审核增强（5 子功能，待建）
- 模块 F：外部数据检索（3 子功能，待建）
- 模块 G：高级分析（5 子功能，远期）
- 模块 H：基础设施（6 子功能，已上线）

**价值**：从「22 个零散优化点」变为「8 个有边界的模块」，开发可以按模块分配。

### 10. 数据模型（§八）—— 从类型片段变为完整类型系统

旧 spec：类型定义散布在 optimization-roadmap 的第六节（仅方向性参考）。

PRD：集中定义了 15 个完整 interface + AppContext 的完整 reducer action union type（含 🆕 标记的新字段），包括：
- 枚举类型（4 个）
- 请求/响应类型（5 个）
- VariantMeta 元数据类型
- 竞品/HK 帖文/话题日历/专家评分/发布策略类型（5 个外部数据类型）
- AppAction 联合类型（20+ action types）

### 11. Prompt 工程架构（§十）—— 从策略描述变为可执行伪代码

旧 spec：5 Layer 分层概念 + 注入格式示例。

PRD 新增：**`buildWritingFramework()` 分支逻辑伪代码**，展示 toggle=ON 和 toggle=OFF 时的完整调用链差异。

## 三、总结：PRD/SDD 让项目从「可理解」变为「可执行」

| 维度 | 旧 comprehensive-spec | 新 PRD/SDD |
|------|----------------------|-----------|
| **读者** | 产品决策者、技术负责人 | + 开发者、测试人员、运维 |
| **用户理解** | 无用户画像 | 2 个具体画像 + 3 个叙事场景 |
| **流程** | 文字描述 | 3 个 Mermaid 流程图（主/异常/辅助） |
| **交互细节** | 散布在 prompt 描述中 | 表格化：交互/状态/边界各 4-6 行 |
| **UI 文案** | 无 | 30+ 条精确文案 |
| **功能组织** | 22 项线性列表 | 8 模块树状结构 + 优先级 |
| **类型系统** | 方向性片段 | 15 个完整 interface + reducer actions |
| **错误处理** | 各接口自由发挥 | 统一 ApiError 6 类错误码 |
| **非功能需求** | 无 | 性能/权限/兼容/安全/存储 5 类 |
| **待决策项** | 分散各处 | 汇总 6 个问题 |
