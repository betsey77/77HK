# 🔍 RAG 优化 — 产品规格书（低优先级 / 后续待开发）

> 撰写日期：2026-07-10
> 关联文档：`comprehensive-spec-v2.md` § P3.1（参考语料库 + RAG 检索）、§ 十（参考语料库设计）
> 功能状态：⏸️ 延后 — Phase 3 远期项目，不在近期版本范围内
> 优先级：🟢 低（当前关键词匹配已满足 <500 条语料规模）

---

## 一、当前状态

### 1.1 已有基础

| 已有能力 | 实现方式 | 数据量 |
|---------|---------|:------:|
| YouTube 热门/搜索 | `youtubeSearchService.ts` — YT Data API v3 | 按需，每次 6-12 条 |
| 竞品广告检索 | `competitorService.ts` — Meta Ad Library + demo fallback | 按需，每次 ≤10 条 |
| 灵感面板展示 | 4 Tab 实时拉取，不缓存 | 每次请求无持久化 |

### 1.2 当前局限

- **无持久化语料库**：每次刷新灵感面板都重新请求 API，之前的数据被丢弃
- **无检索能力**：无法根据「行业 + 平台 + 帖文类型」搜索历史语料
- **无语义理解**：关键词匹配不理解「中秋」和「節日優惠」之间的关系
- **无 RAG**：生成 Prompt 中的 few-shot 示例来自 API 实时结果，不是精选语料

### 1.3 为什么延后

| 原因 | 说明 |
|------|------|
| **数据量不足** | 当前语料通过 API 按需拉取，未持久化。RAG 需要 ≥500 条精选语料才有意义 |
| **关键词匹配够用** | 在 <500 条规模下，倒排索引的关键词匹配与向量检索的差异不大 |
| **更紧急的需求** | P2.10-P2.12（收藏库增强、正例反馈、进度条）对用户体验的提升更直接 |
| **基础设施开销** | Embedding 模型 + 向量存储 + 检索服务 = 显著的复杂度提升 |

## 二、目标架构

### 2.1 RAG 增强后的架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     RAG-Enhanced Generation                      │
│                                                                  │
│  用户输入原文                                                      │
│      │                                                           │
│      ├──→ 关键词提取（行业、产品类型、目标受众、平台）               │
│      │                                                           │
│      ├──→ 语料库检索 ──→ Top-K 相关帖文（K=5）                    │
│      │       │                                                   │
│      │       ├── 稀疏检索：BM25 关键词匹配（行业 + 平台）           │
│      │       └── 稠密检索：Embedding 语义相似度（内容理解）          │
│      │                                                           │
│      ├──→ 重排序（Reranker）                                      │
│      │       ├── 互动率加权（engagement 越高越靠前）               │
│      │       ├── 时效性衰减（30 天内的帖文权重更高）                │
│      │       └── 多样性过滤（避免 5 条全是同一种 Hook 类型）        │
│      │                                                           │
│      └──→ Prompt 注入（Layer 3 Contextual）                       │
│              └── 以 Few-Shot 风格参考格式注入 Top-3               │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 技术选型

| 组件 | 推荐方案 | 备选方案 |
|------|---------|---------|
| **Embedding 模型** | `text-embedding-3-small` (OpenAI) | `bge-small-zh` (本地) |
| **向量存储** | SQLite + `sqlite-vec` 扩展 | Chroma (轻量级向量 DB) |
| **关键词检索** | 内存倒排索引（自定义） | SQLite FTS5 |
| **Reranker** | 无（初版用规则排序） | `bge-reranker-v2-m3` |
| **语料采集** | 定时脚本（Node.js cron） | n8n / GitHub Actions |

### 2.3 语料库规模预估

| 来源 | 采集频率 | 月增量 | 年总量 |
|------|---------|:------:|:------:|
| YouTube 热门 HK | 每日 1 次 | ~90 条 | ~1,000 条 |
| YouTube 搜索（食品/美容/零售） | 每日 1 次 | ~180 条 | ~2,000 条 |
| 竞品广告（按用户搜索触发） | 按需 | ~50 条 | ~600 条 |
| 人工精选标注 | 每月 | ~20 条 | ~240 条 |
| **合计** | | **~340 条/月** | **~4,000 条/年** |

> ⚠️ 至少需要 2-3 个月的自然积累才能达到 RAG 有意义的语料量（~500-1,000 条）。

## 三、分阶段实施计划

### Phase 3a — 语料持久化（无 RAG）

> 目标：每次 API 请求的结果写入 SQLite，建立最基本的语料库。
> 优先级：🟡 中
> 时机：语料量积累到 ~200 条后开始

**任务**：

| # | 任务 | 描述 |
|---|------|------|
| A1 | 创建 `corpus.db` SQLite 数据库 | 表：`posts (id, platform, industry, postType, body, hashtags, engagement_json, url, collected_at, source)` |
| A2 | 修改 `youtubeSearchService.ts` | 每次请求后自动写入 DB |
| A3 | 修改 `competitorService.ts` | 每次请求后自动写入 DB（非 demo 数据） |
| A4 | 创建 `GET /api/corpus/stats` | 返回语料库统计（总量、来源分布、行业分布） |
| A5 | 创建 `GET /api/corpus/search?industry=&platform=&q=` | 关键词搜索（SQLite FTS5 或 LIKE） |

### Phase 3b — 稀疏检索（BM25 / FTS5）

> 目标：用户可以根据行业、平台、关键词搜索历史语料。
> 优先级：🟡 中
> 时机：语料量 ~200-500 条

**任务**：

| # | 任务 | 描述 |
|---|------|------|
| B1 | 启用 SQLite FTS5 全文索引 | `CREATE VIRTUAL TABLE posts_fts USING fts5(body, hashtags, ...)` |
| B2 | 创建 `corpusService.ts` 检索服务 | 封装 FTS5 查询 + 行业/平台过滤 + 互动率排序 |
| B3 | 灵感面板「当下语感」Tab | 新增「搜索语料库」输入框，可搜索历史积累的优质帖文 |
| B4 | Prompt 注入 | 生成时自动检索同行业 Top-3 高互动帖文作为 few-shot |

### Phase 3c — 稠密检索（Embedding + 向量）

> 目标：语义级别的语料检索，理解「中秋」和「節日優惠」之间的关系。
> 优先级：🟢 低
> 时机：语料量 ≥500 条

**任务**：

| # | 任务 | 描述 |
|---|------|------|
| C1 | 集成 `text-embedding-3-small` | 为每条语料生成 1536-d embedding |
| C2 | 集成 `sqlite-vec` | 向量存储 + ANN 近邻搜索 |
| C3 | 混合检索（Hybrid Search） | BM25 + Vector，加权融合 |
| C4 | 创建 `rerankerService.ts` | 互动率 + 时效性 + 多样性重排序 |
| C5 | 升级 `GET /api/corpus/search` | 支持 `mode=hybrid`，返回语义相关 + 关键词匹配的融合结果 |
| C6 | A/B 测试 | 对比「关键词匹配」vs「混合检索」的生成质量评分 |

### Phase 3d — 语料自动采集管道

> 目标：定时自动采集，无需人工干预。
> 优先级：🟢 低
> 时机：Phase 3c 稳定后

**任务**：

| # | 任务 | 描述 |
|---|------|------|
| D1 | 创建 `scripts/collect-corpus.ts` | 每日定时运行（YouTube trending + 多行业搜索） |
| D2 | 重复检测 | 基于 URL/ID 去重，避免重复采集 |
| D3 | 质量过滤 | 低于 100 互动的帖文不录入语料库 |
| D4 | 自动标注 | AI 自动标注行业、帖文类型、Hook 类型、CTA 风格 |
| D5 | 语料健康度监控 | 行业分布均衡性、时效性、来源多样性 |

## 四、RAG Prompt 注入格式（设计稿）

> 当语料库足够大且检索质量稳定后，使用以下格式注入：

```
## 📚 精選參考語料（RAG · 基於「{industry}」行業 +「{platform}」平台）

以下係從語料庫中檢索到嘅同你行業/平台最相關嘅近期高互動帖文。
請參考佢哋嘅表達方式——**學技法，唔好抄內容**：

### 語料 1（IG · 食品 · 產品介紹 · ❤️ 2.3K · 2026-07-05）
> 「是但問個港島人，中秋必買嘅餅舖係？留言話我知👇」

**AI 分析**：
- Hook：反問式 → 即刻製造好奇心 + 互動意圖
- 句式：短句 + 口語節奏，似朋友傾偈
- CTA：直接要求留言，零摩擦
- Emoji：1 個（👇），精準引導行動
- 匹配原因：同行業（食品）+ 同平台（IG）+ 相似目標（節日促銷）

### 語料 2（FB · 食品 · 品牌故事 · ❤️ 1.2K · 2026-07-02）
> 「三十年老字號，由阿爺嗰代開始，我哋堅持每日凌晨四點起身搓粉...」

**AI 分析**：
- Hook：時間線開頭 → 建立信任
- 結構：歷史 → 工藝 → 情感連結
- 語氣：溫暖、真誠、不加修飾
- 匹配原因：同行業（食品）+ 同語氣模式（溫暖真誠）

---
**以上語料來自語料庫檢索（基於 semantic similarity + 互動率排名）。
請在生成時參考以上技法，但確保文案為你嘅品牌/產品原創。**
```

## 五、基础设施与成本

### 5.1 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 数据库 | SQLite (better-sqlite3) | 零配置、嵌入式、Node.js 原生支持 |
| 全文索引 | SQLite FTS5 | 内置，无需额外部署 |
| 向量存储 | sqlite-vec | SQLite 向量扩展，支持 ANN |
| Embedding | OpenAI `text-embedding-3-small` | $0.02 / 1M tokens, 1536-d |
| 定时任务 | node-cron | 轻量，无需外部依赖 |
| 检索服务 | 自研 `corpusService.ts` | 封装在 Express server 内 |

### 5.2 成本预估

| 项目 | 月成本 | 说明 |
|------|:------:|------|
| Embedding 生成 | ~$0.01 | 每月 ~340 条新语料 × ~200 tokens/条 |
| SQLite 存储 | $0 | 本地文件，~10MB/年 |
| 定时采集（YT API） | $0 | 每日配额内（10,000 units/day） |
| **合计** | **~$0.01/月** | 几乎零成本 |

## 六、决策记录

| # | 决策 | 理由 | 可反转 |
|---|------|------|:------:|
| D-RAG-1 | 先做持久化，不做 RAG | 数据量不够，RAG 收益不显著 | ✅ |
| D-RAG-2 | 使用 SQLite 而非向量数据库 | 减少部署复杂度；<1,000 条数据 FTS5 足够 | ✅ |
| D-RAG-3 | 先用 OpenAI Embedding，不本地部署 | 零运维开销；量小成本极低 | ✅ |
| D-RAG-4 | 语料自动采集排在最后 | 手动 + API 按需已满足当前需求 | ✅ |
