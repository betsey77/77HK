# 2.1 Slice E：Bad Case 诊断审阅包开发规格

状态：E0 规格完成，尚未开始业务代码、Migration 或远端操作

日期：2026-07-22

前置基线：1.1.4.5 Slice D7 staging 已通过；现有 D6c 低分详情继续可用

发布边界：本文件只定义下一版；部署、Migration、commit、push 均需独立授权

## 1. 目标与非目标

### 1.1 目标

把当前“低分任务详情”升级为可闭环的 Bad Case 诊断审阅包。每个审阅包要让审核者在同一处回答：

1. 哪个样本出了什么问题；
2. 问题发生在生成链路的哪个阶段；
3. 数据属于谁、问题由哪个内部团队或人员负责；
4. 用哪条可执行的验收标准判断通过或失败；
5. 当时实际使用了哪个规则、知识快照、Prompt 模板和模型策略；
6. 建议如何修改，以及修改后用什么样本复验；
7. 人工审核者最终接受、驳回还是标记误报。
8. 用户可从工作台“账户与更多选项”查看已经真实上线的版本更新日志。

### 1.2 非目标

- 不保存或展示模型隐藏思维链（chain-of-thought）。Trace 只表示业务与模型调用的可观测事件。
- 不把供应商原始请求、原始响应错误、API Key、JWT、Cookie 或环境变量写入日志。
- 不允许 AI 自动发布规则、知识或 Prompt 修改；只能生成待审阅提案和 diff。
- 不把用户正文自动晋升为共享知识或黄金样本；必须经过脱敏、授权和人工批准。
- 不在本切片中部署、执行 Migration、commit 或 push。
- 不把开发中、仅本地完成或仅 staging 验收的功能写成“已上线”。

## 2. 现状与缺口

当前 D6c 已有：完整任务 UUID、原文、五类文案、诊断、审核、评分、错误摘要，以及按 `job_id` 查询的脱敏模型 attempt。详情遵守 `存在/范围检查 -> 强制审计 -> 正文读取`，模型日志缺表只降级日志区。

当前缺少：

- 内部问题责任人、处置状态和审核意见；
- 统一的错误位置与问题分类；
- 可机读、可逐项验收的标准与证据；
- 生成当时的 Prompt/规则/知识/模型策略版本快照；
- 从问题到修复提案、回归样本、人工结论的闭环；
- 复发率、误报率、定位耗时和版本对比指标。

## 3. 产品模型

### 3.1 审阅包组成

每个 `generation_job` 最多对应一个主审阅包，包含以下六个区块。

#### A. 样本 Sample

- 输入：原文、品牌/产品、结构化 brief、卖点、品牌红线和生成设置快照。
- 输出：五个平台变体、诊断、审核、分项分数、消费者反馈和任务错误摘要。
- 定位：每个 finding 指向 `source`、某个 variant、某个评分维度或某条调用事件。
- 隐私：样本正文只在已审计的详情请求中返回，不进入列表、聚合指标或普通日志。

#### B. 运行轨迹 Operational Trace

- 事件字段：阶段、开始/结束时间、状态、attempt、provider/model、耗时、Token、有限错误类别、fallback/retry 关系。
- 建议阶段：`request_validated`、`context_resolved`、`diagnose_generate`、`audit`、`consumer_simulation`、`persisted`。
- `requestId/jobId` 可用于关联，但不得保存供应商原始 payload。
- Trace 明确显示“没有采集”“旧任务不可追溯”和“采集失败”，不得用空数组伪装成功。

#### C. 双 Owner

- `subjectOwner`：数据所属用户，只显示 owner UUID、display name、review group；MVP 不在列表显示邮箱。
- `caseOwner`：内部问题责任人，由 `ownerTeam` 加可空 `assigneeId` 构成。
- 推荐责任团队枚举：`content_prompt`、`knowledge_rules`、`model_provider`、`backend_platform`、`frontend_experience`、`unassigned`。
- 自动分派只给出推荐团队；具体人员指派和变更必须写审计事件。

#### D. 验收标准 Acceptance Criteria

每条标准至少包含：

- `criterionId`、名称、版本、适用范围；
- 可计算条件或人工检查说明；
- `pass | fail | not_evaluated`；
- 实际值、期望值和证据引用；
- 修复后必须执行的回归样本与验收人。

首批标准覆盖：总分阈值、粤语自然度、繁体一致性、平台适配、品牌安全、事实/卖点约束、五平台完整性、模型链路成功、usage 可用性。不能从现有字段可靠计算的项目必须为 `not_evaluated`。

#### E. 可审阅工件 Reviewable Artifacts

- Prompt：模板 ID、版本、内容哈希、段落清单、参数键清单和经脱敏的渲染预览。
- 规则：规则集 ID、版本、哈希、命中的 rule ID 和当时规则文本快照。
- 知识：案例/知识条目 ID、版本或更新时间、标题、来源类型、被使用的最小正文快照。
- 模型策略：provider、model、重试/fallback 策略版本、温度等已允许的非敏感参数。
- 历史任务若没有生成时快照，标记 `legacy_unavailable`；禁止用当前文件内容冒充历史事实。

#### F. Findings 与修复提案

每条 finding 至少包含：

- 分类、严重度、置信度、受影响阶段/变体；
- 问题描述和证据引用；
- 违反的标准、规则或工件版本；
- 推荐责任团队；
- 建议修改的工件类型；
- 可审阅的 before/after diff 或结构化修改建议；
- 人工结论：`confirmed | false_positive | accepted_risk | needs_data | resolved`。

自动提案不得直接覆盖源文件、知识库或数据库记录。发布前必须经过人工批准、回归测试和独立发布门禁。

### 3.2 自动化时机

1. 生成开始时捕获工件 manifest；成功或失败结束时封存结果。
2. 满足任一触发条件时建立审阅包：总分 `< 50`、生成失败、关键验收标准失败，或管理员手动标记。
3. 同步路径只做确定性提取和标准判断，不增加用户生成等待时间。
4. 可选 AI 分析以显式后台任务运行；失败只标记 `analysis_unavailable`，不影响生成结果或既有审阅包。

## 4. 错误位置与指标口径

### 4.1 两类失败必须分开

- 运行失败：超时、鉴权、限流、供应商错误、解析失败、持久化失败等。现有 success/error rate 只回答这一层。
- 质量失败：事实、港味、平台、品牌安全、卖点、Prompt/知识适配等。由 findings 和验收标准回答。

### 4.2 推荐分类

`input_contract`、`context_resolution`、`prompt_instruction`、`knowledge_retrieval`、`model_transport`、`model_output_schema`、`content_quality`、`compliance`、`persistence`、`ui_presentation`、`evaluation_gap`。

### 4.3 下一版指标

- 按阶段/错误类/模型/Prompt 版本的失败数和占比；
- bad case 复发率、同类问题重复样本数；
- 首次定位时间、首次指派时间、关闭时间；
- finding 人工确认率和误报率；
- 验收标准覆盖率、`not_evaluated` 比例；
- 修复前后同一黄金样本集的通过率和分数差；
- 每个成功任务/确认 finding 的官方 Token；人民币成本仅在存在明确的版本化价格表时计算，不能由余额差额伪造。

## 5. 权限、审计与隐私

### 5.1 MVP 权限

- 列表、详情、工件、owner、finding 和处置操作全部仅 `super_admin`。
- 本版不扩大普通管理员权限。未来若开放普通管理员，必须复用当前 `review_group` owner allowlist，跨组在审计和正文读取前返回 404。
- trusted/service-role 查询不依赖 RLS 授权，服务层必须显式重校验 actor scope。

### 5.2 详情读取顺序

必须保持：

1. 只读 `id + owner_id` 做存在性和范围判断；
2. 写入 `admin_view_bad_case_review_pack` 审计，失败即 500；
3. 重新检查范围；
4. 才读取样本正文、owner、工件和 findings。

列表永不返回正文、邮箱、Prompt 正文或知识正文。

### 5.3 写入与发布

- caseOwner、状态、finding 结论和评论写入必须记录 actor、前后值、reason、request ID。
- 所有客户端 owner/actor/role 字段均不可信；服务端只用认证上下文。
- 提案发布为独立高风险切片，不能由本审阅包 API 直接修改 Prompt 源码或知识库。

### 5.4 保留与删除建议

- 原始 model call trace 沿用 90 天目标。
- 审阅元数据建议保留 12 个月；样本正文沿用 generation job 的删除/保留规则。
- generation job 被软删除后，详情立即不可见；后续保留的统计只能是去标识聚合。
- 真实清理任务、保留期 Migration 和历史回填均需独立授权。

## 6. 建议数据模型

Migration 仅在后续切片起草，E0 不创建表。

### 6.1 `generation_artifact_snapshots`

- `id`, `generation_job_id`, `owner_id`
- `prompt_manifest jsonb`, `rule_manifest jsonb`, `knowledge_manifest jsonb`, `model_policy_manifest jsonb`
- `schema_version`, `captured_at`
- `unique(generation_job_id)`

只保存明确白名单字段。内容快照大小设上限，哈希使用稳定 canonical JSON。

### 6.2 `bad_case_review_packs`

- `id`, `generation_job_id`, `subject_owner_id`
- `trigger_kind`, `status`, `owner_team`, `assignee_id`
- `criteria_version`, `analysis_status`, `summary`
- `created_at`, `updated_at`, `resolved_at`
- `unique(generation_job_id)`

### 6.3 `bad_case_findings`

- `id`, `review_pack_id`, `category`, `severity`, `confidence`
- `stage`, `variant_key`, `description`
- `evidence_refs jsonb`, `criterion_refs jsonb`, `artifact_refs jsonb`
- `suggestion jsonb`, `recommended_owner_team`
- `disposition`, `reviewer_comment`, `reviewed_by`, `reviewed_at`

### 6.4 `bad_case_review_events`

追加式事件：状态、指派、人工结论和提案评审。禁止 UPDATE/DELETE 历史事件；只允许受信任 BFF 写入。

### 6.5 访问控制

- `anon/authenticated` 不直接访问以上四表。
- `service_role` 最小表权限；BFF 按 actor 业务权限控制。
- Migration 必须包含 grants、RLS/ACL、索引、JSON 大小约束和反序列化白名单测试。

## 7. API 草案

- `GET /api/admin/bad-case-review-packs`：仅元数据、分类、状态、责任团队和分数；分页/筛选。
- `GET /api/admin/bad-case-review-packs/:id`：审计后返回完整审阅包。
- `POST /api/admin/bad-case-review-packs/:id/assign`：指派责任团队/人员。
- `POST /api/admin/bad-case-review-packs/:id/status`：状态流转。
- `POST /api/admin/bad-case-findings/:id/review`：人工确认、误报、需数据或接受风险。
- `POST /api/admin/bad-case-review-packs/:id/analyze`：显式请求后台 AI 丰富；幂等、限额、失败可重试。
- `POST /api/admin/bad-case-findings/:id/proposal`：创建待审提案，不发布。

错误语义：400 输入错误；401 未登录；403 非超级管理员；404 不存在/不可见；409 非法状态流转；429 分析限流；500 脱敏内部错误。

## 8. UI 信息架构

详情采用标签页或可折叠区域：

1. 概览：严重度、状态、触发原因、双 Owner、关键验收结果；
2. 样本：输入、五平台输出、评分与反馈；
3. Trace：按阶段时间线和 attempt 展示“错误在哪里”；
4. 验收：逐条标准、实际值、证据和复验要求；
5. 工件：Prompt/规则/知识/模型策略版本与可审阅差异；
6. Findings：人工处置与待审提案；
7. 审计：只显示业务事件，不显示 secrets。

桌面和 390px 均不得横向溢出；宽表在组件内部滚动。正文和工件默认折叠，避免一次加载/渲染全部大字段。

### 8.1 工作台更新日志

- 入口位于工作台右上角 `HeaderMenu`，名称为“更新日志”；打开独立 drawer/dialog，不跳离当前创作状态。
- 首版使用代码内静态、可版本审阅的数据源，不新增数据库、RLS 或公开 API。
- 每个发布记录包含版本号、生产上线日期、用户可见功能、体验优化、重要修复和必要的使用提示。
- 只展示生产环境已实际部署并完成验收的内容。`local`、`staging`、`preview`、待迁移和待人工验收项目不得进入已发布列表。
- `2.1` 的完整条目在 E9 人工验收、生产部署成功并核对部署 manifest 后一次性写入，覆盖本次真正上线的全部用户可见更新。
- 不暴露内部 UUID、review group、供应商密钥、管理员安全实现、数据库结构或尚未公开的内部功能。
- drawer/dialog 必须支持 Escape、焦点管理、body scroll lock、桌面/390px 内滚动和空状态；不增加未读 badge、推送或账号级已读状态。

## 9. 实施切片与验收门禁

### E0 — 规格与执行编排（本次）

- 产物：PRD/SDD/TEST_PLAN/ACCEPTANCE、本文和 Grok 执行模板。
- 验收：术语、权限、数据模型、切片、工作区基线门禁一致；无业务代码和远端操作。

### E1 — 工件版本与快照合同

- 定义 Prompt/规则/知识/模型策略 manifest、版本和稳定哈希。
- 新任务在生成时捕获实际使用工件；旧任务明确 `legacy_unavailable`。
- 验收：双模型路径、案例知识、卖点/红线与规则版本均有契约测试；禁止 secrets/CoT。

### E2 — Review Pack 本地数据层与自动触发

- 起草四表 Migration、类型、repository 和确定性触发器/组装器。
- 验收：`<50`、失败、关键标准失败、手动标记均幂等创建一个 pack；Migration 仅本地。

### E3 — 审计详情 API、Owner 与 Trace

- 实现列表/详情，双 Owner、阶段时间线和有限错误分类。
- 验收：非 super_admin 403；详情严格 `scope -> audit -> recheck -> body`；审计失败 fail-closed；列表无正文。

### E4 — 标准引擎与 Findings

- 实现版本化标准、证据引用、自动分类和责任团队建议。
- 验收：每个 fail 都有证据；未知数据为 `not_evaluated`；不生成无来源断言。

### E5 — 工件审阅与修复提案

- 展示 Prompt/规则/知识快照和 before/after 提案。
- 验收：历史缺失不冒充；提案不能直接发布；敏感字段静态和运行时双重拒绝。

### E6 — 管理端闭环 UI

- 实现筛选、详情标签页、指派、状态和人工 disposition。
- 同一切片增加工作台 `HeaderMenu -> 更新日志` 的静态 drawer/dialog 外壳，但不得提前发布 `2.1` 已上线条目。
- 验收：桌面/390px、键盘、加载/空/错误/降级态、超长正文与内部滚动均通过；更新日志不破坏当前工作台状态。

### E7 — 评测样本候选与诊断指标

- 经人工批准把脱敏样本加入 evaluation candidate；展示分类、复发、误报、覆盖和解决时长。
- 验收：未经批准不晋升；owner 内容不进入跨用户共享集；价格表缺失时不显示人民币估算。

### E8 — Staging 数据与安全门禁

- 经单独授权执行 dry-run、单次 Migration、RLS/ACL/并发/审计/回滚/清理验证。
- 验收：无残留 QA 数据，Advisor 复核，真实 super_admin/普通 admin/跨 owner 路径通过。

### E9 — 人工验收与发布准备

- 用户最终核验业务闭环；更新部署手册与回滚检查表。
- 生产部署成功后，按最终部署 manifest 将本次所有已上线用户可见更新写入 `2.1` 更新日志；任何未随本次部署上线的项目必须排除。
- 只有人工验收后，才可另行授权 commit/push/Vercel Preview/Promote。

## 10. Grok Build CLI 并行执行协议

本节只约束后续实施；E0 不启动 Grok，也不创建 worktree。

### 10.1 机制边界

- 只使用 Grok CLI 自身的 Agent Teams / Autonomous Agents / worktree / background task；不创建 Codex 子代理。
- 当前可执行文件为 `grok 0.2.106`。实际入口是 `grok`，已确认支持 `--agent`、`--agents`、`--worktree`、`--worktree-ref`、`--permission-mode`、`--max-turns` 和 headless 模式；不存在需假设的 `grok build` 子命令。
- 一个 leader 最多分派三个一级成员；成员禁止再次创建 task、subagent 或 team。
- 若当前 CLI 不能可靠限制为一层，停止 Team 模式，退化为每个 worktree 一个独立 leader；不得仅靠口头提示假装隔离成功。

### 10.2 权限

- leader 与所有成员使用同一 `--permission-mode auto` 和同一 allow/deny 规则，不给成员另设更弱或更强权限。
- 禁止 `bypassPermissions`、`--always-approve` 和无约束 YOLO。
- deny 至少覆盖：读取 secrets、修改 `.env`、数据库 push/migration、部署、commit/push、`git reset`、`git clean`、递归删除和 worktree 删除。
- 成员只拥有自己文件域；共享路由入口、公共类型和规格由 leader 单独合并。

### 10.3 当前 Dirty Worktree 基线闸门

当前 HEAD 不包含大量已完成但未提交的 D1-D7 和其他本地改动。直接从 HEAD 使用 `--worktree` 会得到过期代码，漏掉当前真实基线。

在用户批准以下任一可复现方案前，Grok worktree 模式为 **BLOCKED**：

1. 用户授权创建包含当前目标基线的 checkpoint commit/ref；或
2. 用户批准另一种经过完整文件清单、秘密扫描和可重放验证的基线同步方案。

不得静默复制整个 Dirty Worktree、不得把未跟踪 secrets 带入 worktree、不得为了创建 worktree 执行 reset/clean。闸门解除后，每个工作树必须显式使用同一 `--worktree-ref <approved-ref>`。

2026-07-22 闸门已解除：用户授权后已用独立临时 Git index 创建本地 `refs/codex/checkpoints/2026-07-22-slice-e-baseline`；`master`、真实暂存区和 Dirty Worktree 均保持不变。后续 Grok worktree 必须显式从该 ref 创建，不得退回过期 HEAD。

### 10.4 推荐并行通道

- Lane A：E2 Migration/RLS/ACL 合同，仅本地草案；
- Lane B：E3/E4 server API、Trace 和标准引擎；
- Lane C：E6 client UI，先使用冻结 API fixture；
- Lane D：E1/E5 工件 manifest、提案模型和专项测试。

依赖顺序：E1 合同冻结 -> E2 数据接口冻结 -> B/C/D 可并行 -> leader 集成 -> 全量验证。两个 lane 不得同时修改同一文件。

### 10.5 后台任务与停止条件

- 后台任务只用于独立 worktree 的实现、只读搜索或测试；数据库、部署和 Git 发布动作不得后台执行。
- 每个任务输出结构化摘要：改动文件、测试、风险、未决项、基线 ref 和 worktree 名称。
- 每个任务最多 12 turns；相同容量/登录错误两次无新证据即停止；同一实现问题最多三次尝试。
- Grok 结果始终是候选改动；主代理必须复核 diff、权限边界和测试后才能进入下一切片。
