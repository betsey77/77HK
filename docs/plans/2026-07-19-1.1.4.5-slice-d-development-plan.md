# 1.1.4.5 Slice D 详细开发计划

日期：2026-07-19  
状态：D0-D4 本地实现完成；D1/D4 Migration 均未应用  
范围：连续签到、会员奖励、运营指标、模型健康与 Token 用量

> 本文中的 Slice D 指 `1.1.4.5` 新增功能，不是 2026-07-12 已完成的“收藏/品牌/配置云同步 Slice D”。后续任务、证据目录和测试名称统一使用 `1.1.4.5 Slice D` 或 D0-D7，避免混淆。

## 1. 本切片目标与非目标

目标：

1. 用户按香港自然日签到；连续 7 天后获得一次 30 天 Pro 奖励，整个过程可审计、可幂等、不可由浏览器直接改会员。
2. 普通管理员只能查看当前授权组的活跃、会员发放和额度指标；`super_admin` 才能查看全局及模型健康。
3. 记录模型调用成功/失败、耗时和供应商返回的 Token 用量，同时不保存 Prompt、回复正文、密钥、JWT 或原始供应商错误。
4. 复用现有 `subscriptions` 和 `usage_ledger` 作为额度事实源，不建立第二套额度账本。

非目标：

- D0 当时不实现业务代码；当前已完成 D1-D3，D4-D7 仍不在本切片内。
- 未经单独授权，不创建或推送 Migration，不连接 staging 写数据，不部署。
- 第一版不硬编码模型价格或估算资金余额；价格会变化，供应商余额只使用官方接口的真实结果。
- 不引入 Realtime、消息队列、定时任务平台或大范围服务重构。

## 3. 推荐业务规则

### 3.1 签到

- 日期只由服务端/数据库按 `Asia/Hong_Kong` 计算，客户端日期不参与判定。
- 同一用户同一香港日期只有一条签到记录；重复点击返回同一结果，不重复增加天数。
- 当天与上次签到相邻一天时连续天数 `+1`；间隔超过一天则从 `1` 开始。
- 第 7 天在同一事务中创建唯一奖励记录。并发请求依靠行锁和唯一约束收敛为一个结果。
- 每日首次进入已登录网站时展示签到入口；用户关闭后只在本机保存“当天已关闭”，但连续天数和奖励状态始终以服务端为准。

### 3.2 奖励

- Free 或没有有效 Pro 的用户：第 7 天立即创建 30 天 Pro 周期，`quota_used=0`，奖励标记为 `applied`。
- 已有有效 Pro 的用户：只创建 `pending` 奖励，不修改现有订阅结束时间和额度；到期后用户主动领取，领取接口再次加锁并检查当前状态。
- 用户在待领取期间购买或获开通新的 Pro，奖励继续保持待领取，直到没有有效 Pro 才可领取。
- 会员写入与奖励审计必须在同一受信任事务中完成；失败全部回滚，不允许“已显示领奖但未开通会员”。

## 4. 数据设计

### 4.1 `daily_checkins`

- `id`, `user_id`, `checkin_date_hk`, `streak_count`, `streak_key`, `created_at`
- 唯一约束：`(user_id, checkin_date_hk)`
- 用户只能读取自己的记录；浏览器和普通登录角色均不能直接插入、更新或删除。

### 4.2 `membership_grants`

- `id`, `user_id`, `source`, `source_ref`, `duration_days`, `status`, `applied_at`, `subscription_id`, `created_at`
- `source='checkin_7day'`，若采用终身一次奖励，则唯一约束为 `(user_id, source)`。
- `status` 第一版只需要 `pending`、`applied`；不预先设计转赠、撤回等未要求状态。
- 用户只读取自己的安全字段；会员写入只能由服务端受控事务执行。

### 4.3 `app_activity_daily`

- `user_id`, `activity_date_hk`, `first_seen_at`, `last_seen_at`
- 主键：`(user_id, activity_date_hk)`；同一用户一天只贡献一个活跃人数。
- 组指标查询时按当前 `profiles.review_group` 关联。这样普通管理员永远只能看到当前授权组；代价是用户换组后历史聚合会随当前分组变化，第一版接受这个语义。

### 4.4 `model_call_logs`

最小字段：

- `id`, `created_at`, `job_id`（可空、删除任务时置空）、`request_id`（不透明随机值）
- `operation`, `provider`, `model`, `status`, `error_class`, `latency_ms`, `attempt`
- `prompt_tokens`, `completion_tokens`, `total_tokens`, `cache_hit_tokens`, `cache_miss_tokens`, `usage_source`

隐私约束：

- 不保存 Prompt、生成正文、原始错误文本、邮箱、JWT、API Key。
- 模型健康只供 `super_admin` 使用，日志本身不需要复制 `user_id` 或 `review_group`。
- bad case 通过受控查询关联 `generation_jobs` 的既有评分，不在模型日志重复保存正文或评分副本。

## 5. API 与服务边界

用户接口：

- `GET /api/me/check-in`：返回今日是否签到、连续天数、奖励状态、是否可领取。
- `POST /api/me/check-in`：执行幂等签到并返回同一状态模型。
- `POST /api/me/membership-grants/:id/claim`：仅领取当前用户的待领取奖励。
- `POST /api/me/activity`：登录后每个会话/自然日轻量上报一次；服务端按香港日期 upsert。

管理接口：

- `GET /api/admin/metrics/overview?from=&to=`：活跃、会员发放、额度消耗/结余。普通管理员强制当前非空 `review_group`，超级管理员可看全局。
- `GET /api/admin/metrics/models?from=&to=`：仅 `super_admin`，返回成功率、错误率、耗时、Token 聚合、供应商和模型。
- `GET /api/admin/metrics/bad-cases?from=&to=`：仅 `super_admin`，返回低于阈值的受限任务摘要，不直接返回完整正文。
- `GET /api/admin/metrics/provider-balance`：仅 `super_admin`；服务端调用 DeepSeek 官方 `GET /user/balance`，缓存 5-15 分钟。失败显示“暂不可用”，不能显示为 0，也不能返回密钥。

所有时间范围限制在最多 90 天。现有 `/api/admin/stats` 是全局口径，D6 必须先完成权限/口径审计，不能直接把它当普通管理员的新指标接口。

## 6. 模型调用埋点策略

- 在 DeepSeek/CantoneseLLM 服务边界记录每一次真实尝试，包括重试和 fallback；最终生成任务状态继续以 `generation_jobs` 为准。
- 从供应商响应的 `usage` 规范化 prompt/completion/total/cache hit/cache miss Token。没有返回的字段保持 `null`，不猜测。
- 第一版使用显式可选 `ModelCallContext` 从路由传入服务，而不是全局 AsyncLocalStorage。显式参数改动更可见、测试更直接，也避免并发上下文串线。
- 日志写入采用“等待但不阻断业务”的短超时 best-effort：写日志失败不应让生成失败，同时避免 serverless 中 fire-and-forget 丢失。
- 第一版展示 Token 和官方余额；不内置供应商价格表，不把估算成本冒充账单。

## 8. Grok Build 使用规则

用户已允许在需要时使用 Grok Build，包括 `grok build1` 的 agent 模式。为保护当前 Dirty Worktree：

1. 每次只给 Grok 一个 D1-D7 子切片，并写明可读/可改文件白名单、非目标、测试与停止条件。
2. 代码任务最多 12 个 turn；连续两次无新证据或同一问题三次失败立即停止。
3. Grok 完成后由 Codex 检查前后 `git status`、逐行 diff、敏感字段与测试结果；Grok 结论不能替代验收。
4. `grok agent` 是供 SDK/客户端连接的长驻服务。只有当前流程具备连接客户端、能够限制工具和文件时才启动；普通单次编码优先使用有界 headless 调用。
5. 不创建 worktree，不 reset/clean，不读取或输出密钥，不部署、迁移、commit 或 push。
